# M014: Puzzle Controls & Layout Redesign — Context

**Gathered:** 2026-03-21
**Status:** Ready for planning

## Project Description

SimpleGame is a mobile jigsaw puzzle game built on Unity. Puzzle pieces are generated by the `simple-jigsaw` package (`Packages/simple-jigsaw/`), which produces meshes in a normalized 0–1 board space. The InGame scene currently renders pieces as world-space 3D GameObjects scaled to fit screen, with a static orthographic camera and tray pieces also in world space tapped via `OnMouseDown`.

## Why This Milestone

Two problems:
1. The package's normalized 0–1 sizing makes piece scale depend on grid dimensions — a piece in a 4×2 grid is 0.25×0.5 in local space. Downstream code compensates with complex scaling math. Fixing this at the source gives all consumers a predictable 1-unit-longest-edge contract.
2. The InGame layout uses a static camera. On mobile, a panning camera over the board is the standard jigsaw UX. The tray needs to follow the camera, and slot input needs to not conflict with board pan gestures.

## User-Visible Outcome

When this milestone is complete:
- Pieces are always sized so their longest edge = 1 world unit, regardless of grid dimensions
- The puzzle board renders in world space at natural scale (no scaling transform on the parent)
- The player can drag on the board to pan the camera and navigate the puzzle
- The tray sits at the bottom of the screen and follows the camera pan
- Tray slots have UGUI Button overlays — tap (pointer-up-inside) places the piece with full tween animation
- A hint surface (solved piece outlines) is rendered behind the board pieces

## Implementation Decisions

- **Piece sizing formula:** `longestNorm = Mathf.Max(1f/cols, 1f/rows)`. `unitScale = 1f / longestNorm`. `cellW = unitScale / cols`, `cellH = unitScale / rows`. For a 4×2 grid: unitScale=2, cellW=0.5, cellH=1. Board = 2 wide × 2 tall.
- **Board parent:** `scale = (1,1,1)`, positioned at world origin. `InGameSceneController` drops all `boardSize` scaling math.
- **Camera pan:** `CameraController` MonoBehaviour on Main Camera. Pointer-down on world (no UI hit) → track delta → translate camera XY.
- **Hint surface:** `HintSurfaceBuilder.Build(board.Pieces, thickness: 0.02f)` → MeshFilter+MeshRenderer child of board parent at `z = +0.1f`. Default material.
- **Tray follow:** `LateUpdate` — each frame, tray piece world positions set relative to camera bottom.
- **Slot buttons:** Screen Space Overlay Canvas, one Button per slot. Positioned via world-to-screen projection each frame. `onClick` fires `OnTapPiece`. `PieceTapHandler` removed from tray pieces.
- **Conflict resolution:** UGUI Screen Space Overlay eats pointer events before physics raycasts — button taps never reach camera pan handler naturally.

## Risks and Unknowns

- `GridPlannerTests` has ~6 tests hardcoded to `1f/cols` / `1f/rows` sizing — all need updating.
- `HintSurfaceBuilder` default thickness `0.003f` is normalized-space; with 1-unit pieces use `0.02f`.
- Camera pan unbounded for now — clamping deferred.
